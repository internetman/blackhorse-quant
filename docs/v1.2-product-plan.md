# 黑马自选 v1.2 产品需求文档

## 1. 背景与目标

### 1.1 为什么做 v1.2

- 原「圈子选股 + 意见领袖推荐」的商业模式不成立（大哥不愿承担推荐责任与风险）。
- 产品收缩为：**每人维护自己的关注股票，看 AI 买卖建议与买卖分析、真实行情、重大新闻**，一页看全。

### 1.2 产品方向（一句话）

对**我关注的股票**，在「我的关注」页一次性看到：**真实股价**（最新价、涨跌幅、成交量，涨红跌绿）、**AI 买卖建议**、**买卖分析**（技术面）、**重大新闻动态**；支持添加/取消关注；**复盘记录**保留，用于看 AI 预测准不准、准确率多少。

### 1.3 原则

- **做**：个人关注列表（增删）、真实行情、AI 买卖建议、买卖分析、个股新闻、复盘记录、用户管理（原圈子管理）。
- **不做**：圈子选股、谁推荐、我的持仓；本版本不包含持仓与仓位管理。
- **真实数据**：行情与新闻用 **akshare**（开源，无需注册，直接 `pip install akshare` 调用）；买卖分析接入 stock-arsenal 或同源逻辑。
- **统一口径**：对外股票代码统一使用 `symbol=600519.SH / 000001.SZ`；调用 akshare 时在后端做代码映射，前端不关心数据源差异。

### 3.2.1 添加关注时的股票搜索（模糊输入）

点击「添加关注」后，需提供**模糊输入、联想搜索**的选股方式，行为与通用行情/交易软件一致。

**支持的输入方式**

- **股票代码（部分）**：输入几位代码（如 `600`、`000`、`300519`）即可联想出匹配的股票列表，用户点选添加。
- **股票名称（前几个字）**：输入名称前几个汉字（如「贵州」「宁德」）可匹配并展示候选股票。
- **首字母 / 拼音**：支持拼音首字母（如 `GZMT`）或拼音（如 `guizhou`）联想出对应股票（如贵州茅台）。

**交互与实现要点**

- 输入框带**实时联想**：输入过程中（或防抖后）请求搜索接口，下拉展示候选列表；列表项展示「代码 + 名称」等，点选即添加关注。
- 后端需提供**股票搜索/联想 API**（`GET /api/stocks/search?q=xxx&limit=20`），支持按代码前缀、名称前缀、拼音/首字母模糊匹配；数据源可与 akshare 或本地股票基础表一致。
- 搜索触发建议：`q` 长度 `< 2` 不触发远端查询，避免无效高频请求。

### 3.3 首页 = 我的关注（一页看全）

| 项 | 说明 |
|----|------|
| 路由 | `/recommendations` 为首页（`/` 重定向到此）。 |
| 内容 | 每只关注股票一张卡片，卡片上展示：真实股价（见 3.5）+ AI 买卖建议 + 买卖分析（见 3.6）+ 重大新闻（见 3.7）；可选简单图表（迷你 K 线/分时）。 |
| 数据 | 列表来自用户 watchlist；每只股票数据来自：行情接口 + 分析接口 + 新闻接口（可后端聚合为一条首页 API）。 |
| 导航 | 侧栏/底部仅保留：首页（我的关注）、复盘记录、用户管理（管理员）。 |

### 3.3.1 卡片内容层次与动效规范

AI 买卖分析必须**详细**，但展示上采用**从结论到原因的层次结构**，并在最醒目处给出明确交易推荐。

**一、最醒目位置（结论先行）**

在每张股票卡片的**最上方或最突出区域**，必须清晰展示：

- **今日交易推荐**：买入 / 卖出 / 观望 等明确结论。
- **推荐价格**：今日**买入推荐价格**和/或**卖出推荐价格**（若有），数字要醒目。
- **多空态度**：**看涨**还是**看跌**（或中性），用文案或图标一眼可辨。

以上为「结论层」，用户扫一眼即可得到「今天该怎么操作、什么价位」。

**二、层次结构（结论 → 理由 → 支撑）**

卡片内容按以下顺序组织，形成从结论到原因的阅读路径：

1. **结论层**（最醒目）：今日交易推荐、推荐价格、看涨/看跌（如上）。
2. **理由支撑**：买卖分析的理由（技术面、信号、得分等），AI 买卖分析要写详细，支撑结论。
3. **新闻等支撑**：该股票重大新闻/动态，作为补充依据；可折叠或放在卡片展开区。

**三、内容过长与卡片动效**

- 当理由、新闻等内容较多时，**不得**在首屏堆砌过长文案。
- 卡片需支持**收起 / 展开**：默认展示「结论 + 简短摘要」，详情（详细理由、新闻列表等）置于**可展开区域**。
- **动效**：展开/收起时使用**卡片缩放或展开动效**（如高度过渡、折叠动画），交互清晰、不突兀。

### 3.4 复盘记录（保留）

| 项 | 说明 |
|----|------|
| 路由与页面 | 保留 `/reviews` 及复盘记录页。 |

### 3.5 真实股价信息（必须）

| 项 | 说明 |
|----|------|
| 需求 | 每只关注股票展示**真实行情**；**涨了红色、跌了绿色**（A 股惯例）。 |
| 必须展示 | **最新价**、**涨跌幅**（百分比+颜色）、**成交量**。 |
| 可选 | 简单图表（迷你 K 线、分时图）。 |
| 数据来源 | akshare（开源，无需注册），与行情/新闻统一；非交易时间可返回最近收盘或缓存。 |
| 接口 | `GET /api/quote?symbol=600519.SH` 或合并在首页聚合中，返回 `{ latestPrice, changePercent, volume, ... }`；若数据源仅支持 6 位代码（如 `600519`），由后端完成转换。 |
| 前端规范 | 涨（changePercent > 0）红色，跌（< 0）绿色，持平灰色；数值带正负号或箭头。 |

### 3.6 买卖分析与 stock-arsenal

| 项 | 说明 |
|----|------|
| 需求 | 对每只关注股票给出**买入/卖出建议**（结论、理由、风险）及**买卖分析**（得分、关键指标如 RSI/MACD）。 |
| 数据来源 | stock-arsenal 逻辑（买入信号、跌停预警等）或同源实现；数据源可用 efinance/akshare。 |
| 卡片展示 | 遵循 **3.3.1 卡片内容层次与动效规范**：结论（推荐价、看涨/看跌）最醒目，其次理由与买卖分析（得分、关键指标），跌停预警可合并进 risks 或单独小标签。 |

## 4. 后端 API 约定

### 4.1 全局约定（必须统一）

- **鉴权**：除 `/login` 外接口默认需要登录态；后端从 token 解析 `userId`，前端不得在请求参数中传 `userId`。
- **时区与交易日**：统一使用 `Asia/Shanghai`；“今日建议”按 A 股交易日定义，非交易日展示最近交易日结果并标注日期。
- **symbol 规范**：对外统一 `600519.SH / 000001.SZ`；内部可转换为 `600519` 调 akshare。
- **错误返回**：建议统一 `{ code, message, requestId }`，便于排障与前端降级展示。

### 4.2 接口清单（v1.2 终态）

| 接口 | 说明 |
|------|------|
| `GET /api/watchlist` | 返回当前用户关注列表。 |
| `POST /api/watchlist` | 添加关注，body 至少含 `symbol`；重复添加返回幂等成功或 409（二选一并全局统一）。 |
| `DELETE /api/watchlist/{symbol}` | 取消关注（按当前用户 + symbol 删除）。 |
| `GET /api/stocks/search?q=xxx&limit=20` | 股票搜索/联想；支持代码前缀、名称、拼音/首字母；`q` 长度 `< 2` 可直接返回空列表。 |
| `GET /api/quote?symbol=xxx` | 行情：最新价、涨跌幅、成交量、更新时间。 |
| `GET /api/recommendations` | 首页聚合接口：按当前用户 watchlist 返回每只股票的行情 + 建议 + 分析 + 新闻摘要。 |
| `GET /api/news?symbol=xxx&limit=10` | 个股新闻（akshare `stock_news_em`），或由聚合接口内联返回。 |
| `GET /api/reviews`、`GET /api/reviews/stats` | 复盘列表与准确率（仅当前用户可见）。 |
| `/api/admin/*` | 用户管理（不变，仅管理员可访问）。 |
| `/api/positions` | v1.2 视为废弃接口：保留兼容时返回空数据并标注 deprecated，或直接返回 `410 Gone`（两者选其一并统一）。 |

## 6. 数据与依赖说明

- **akshare**：开源财经数据库，`pip install akshare` 即可，无需注册。用于行情、个股新闻（如 `stock_news_em`）等。
- **stock-arsenal**：买入信号、跌停预警等分析逻辑；若本仓库无完整 `src`，可 subprocess 调脚本或在本仓库实现简化版，数据源与 akshare/efinance 统一。
- **复盘**：建议按用户隔离（只看到自己的历史建议复盘）。

### 6.1 非功能要求（避免上线后高频返工）

- **缓存策略**：行情接口建议交易时段 `15-30s` 短缓存、非交易时段 `5min` 缓存；新闻接口建议 `5min` 缓存。
- **超时与降级**：外部数据源调用超时建议 `2-3s`；某一路失败不应拖垮整页，卡片按模块降级（如“新闻暂不可用”）。
- **性能目标**：`/api/recommendations` 在 20 只关注股票规模下，目标 `P95 < 2s`（命中缓存场景）。
- **可观测性**：关键接口记录 `requestId`、外部源耗时、失败率，便于排查 akshare 或分析脚本抖动。

## 7. 实施顺序建议

1. 后端先统一基础约定：鉴权口径、`symbol` 规范、错误码结构、时区/交易日口径。
2. 后端改造 watchlist（按 userId 隔离 + `DELETE /api/watchlist/{symbol}`）并完成历史数据迁移脚本（旧字段 addedBy* 清理）。
3. 后端实现股票搜索接口（`/api/stocks/search`）与行情/新闻接口（akshare），加缓存与超时控制。
4. 后端实现首页聚合接口 `/api/recommendations`（单接口聚合行情 + 建议 + 分析 + 新闻摘要）。
5. 前端改首页为「我的关注」单页卡片，接入添加/取消关注、联想搜索、卡片展开动效、异常态降级展示。
6. 前端移除 `/watchlist`、`/positions` 相关入口和页面；`/admin` 更名为用户管理语义。
7. 联调与验收：重点验证“用户隔离、symbol 映射、降级策略、复盘准确率口径”后再更新 README 与发布说明。

### 7.1 验收清单（Definition of Done）

- 首页仅展示当前用户关注股票，不出现其他用户数据。
- 行情涨跌配色遵循“涨红跌绿平灰”，并显示最近更新时间。
- 添加关注支持代码/名称/拼音联想；重复添加行为与接口约定一致。
- 关闭或降级 `/positions` 后，前端无死链、无报错弹窗。
- 任一外部源失败时页面可用，且用户能看到明确降级提示。
- 复盘统计与列表仅统计当前用户，且可追溯到 `userId + tradeDate + symbol`。
